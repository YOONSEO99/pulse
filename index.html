Network topology

How traffic actually moves

Every request enters through a single AWS ALB, passes through Nginx Ingress inside the private subnet, and gets split between the stable and canary versions of the app.

NETWORK DESIGN
A single Nginx Ingress tied to one ALB handles all external traffic — no extra load balancers per service. Pods in the private subnet are never exposed to the internet directly. S3 access goes through a VPC Gateway Endpoint, so no traffic ever leaves the AWS backbone.

GitOps & self-healing

The system fixes itself.

Push a commit. Argo CD syncs it. Argo Rollouts deploys it safely. Prometheus watches it. If something breaks, the rollback happens automatically.

Git is the single source of truth
Argo CD constantly watches your Git repo. The moment a commit lands, it syncs the cluster to match — no manual kubectl, no drift.
Canary gets 50% at first
New versions receive 50% of real traffic right away. If something breaks, the rollback kicks in before most users even notice — and only that window of traffic is affected.
Prometheus is the judge
Argo Rollouts queries Prometheus directly via HTTP API every 10 seconds. If any HTTP 5xx error is detected (delta > 0) during the evaluation window, the self-healing protocol triggers an immediate abort.
No human needed for rollback
The entire detect → abort → restore cycle happens in under one second. Stable pods absorb 100% of traffic again before most users even notice something went wrong.
